# MEMORY.md

Durable memory for the process-mining research agent. Append only stable facts, definitions, conventions, and decisions. Do not store transient status here; use `CONTINUITY.md` instead.

## Definitions

- **Event log**: a collection of traces/cases; each trace is an ordered sequence of activity labels, optionally with timestamps and attributes.
- **Process discovery**: learning a process model from an event log.
- **Petri net output**: represented here as transitions, places, directed arcs, an initial marking, and a final marking.
- **PMIR**: Process-Mining Intermediate Representation. A deliberately lightweight intermediate format that can store activities, start/end evidence, direct-follows evidence, inferred relations, block/decomposition hints, and place candidates before compilation to a Petri net.
- **Limited-operation model**: a declared set of primitive operations and a rule for counting them. For the first goal, the default counted primitives are event scan, dictionary increment, comparison, set insert, set lookup, arithmetic, relation classification, and arc/place construction.
- **First-goal soft operation budgets**: for later screening, use `B_smoke = 10N + 8A^2 + 6D + 80` on toy logs and `B_deep = 16N + 12A^2 + 10D + 120` on small synthetic logs. These are not quality metrics and do not override replay, negative probes, or structural diagnostics.

## Durable research principles

- Candidate algorithms must be compared on fitness, precision/generalization behavior, simplicity, soundness/structural properties, robustness, and operation cost.
- A failed algorithm is valuable if it yields a counterexample, limitation, or useful intermediate representation.
- Promising algorithms should be refined through ablations and counterexample-guided repair before formal claims are made.
- Super-promising algorithms require a property dossier before being presented as a serious research result.

## Default smoke-log families

- Strict sequence: `A B C D`
- Exclusive choice: `A (B|C) D`
- Parallel/concurrent behavior: `A B C D` and `A C B D`
- Short loop: `A B A C` or `A B B C`
- Skip behavior: `A B C` and `A C`
- Noise/infrequent behavior: dominant pattern plus rare directly-follows deviations

## Do-not-forget decisions

- The repository must remain generic. The first Petri-net task is a goal, not the agent identity.
- Intermediate representations are allowed and encouraged when they make discovery or property study easier.
- In this workspace, use `python3` for benchmark commands; `python` was not available during EXP-0002.
- Replay fitness from the local token-game evaluator is not a precision or soundness claim. Structural diagnostics and negative-trace probes are required before promotion.
- A candidate with unconstrained visible transitions must not be promoted based on positive-trace replay alone.
- The local token-game evaluator supports bounded silent-transition closure for `tau_` transitions; this is enough for smoke replay of optional skips, not a full reachability/soundness proof.
- Local optional-skip guards can conflict when multiple optional triples overlap. Future PMIR guard compilers need a conflict-resolution or block-detection step before emitting optional places.
- Transitive-reduction optional-chain compilation can repair overlapping optional skips, but it is costlier and does not solve optional behavior embedded in concurrency.
- The cut-limited process-tree baseline (`ALG-0003`) currently handles top-level sequence, common-context XOR, common-context parallel, optional-sequence blocks, and one bounded optional-concurrency form: a mandatory singleton branch parallel with a two-step branch whose second activity is optional. It is not yet recursive and should not be treated as a full inductive miner.
- Petri-net JSON may include optional `transition_labels` for candidates that need multiple visible transitions with the same activity label. The evaluator still infers legacy `t_<activity>` labels when no explicit label is present.
- The prefix-automaton comparator (`ALG-0005`) is an exact observed-language baseline. Its strong replay/negative-probe behavior should be read as memorization until held-out and variant-explosion tests are run.
- EXP-0010 held-out tests confirm `ALG-0005` does not generalize to unobserved valid interleavings and memorizes observed noise. It needs a grammar/block abstraction before any deep-testing promotion.
- The bounded visible-place region comparator (`ALG-0004`) can recover sequence, XOR, and parallel places, but EXP-0011 shows high enumeration cost, optional-skip overgeneralization, and unresolved loop behavior. It needs silent optional-place patterns or a different compiler layer before promotion.
- The region optional-tau repair (`ALG-0011`) fixes non-overlapping singleton optional skips only when a selected shortcut place certifies the skip. EXP-0012 shows this repairs `skip.json` precision versus `ALG-0004`, but overlapping optional chains and optional behavior inside concurrency remain unresolved.
- EXP-0013 confirms `ALG-0011`'s silent optional repair is causal: the no-repair ablation matches `ALG-0004`, while the repair fixes singleton and two-disjoint optional skips. It still rejects overlapping optional-chain repair candidates and does not repair optional behavior inside concurrency.
- The region optional-chain repair (`ALG-0012`) uses selected region shortcut places to certify order-consistent optional chains. EXP-0014 fixes overlapping optional chains versus `ALG-0011`, but costs more than `ALG-0010` and still fails optional behavior inside concurrency.
- EXP-0015 shows the no-certification ablation (`ALG-0013`) currently matches `ALG-0012` on optional-pattern and broader synthetic replay/negative metrics while saving only tiny operation counts. This does not justify replacing `ALG-0012`; it creates a next counterexample-search question about whether selected-region certification ever prevents harmful optional-chain emission.
- The `optional_singleton_parallel_branch` probe is a durable optional/concurrency counterexample: region variants replay positives but overgeneralize one negative, `ALG-0010` loses one positive, and the current `ALG-0003` optional-concurrency cut loses all positives.
- The prefix block abstraction (`ALG-0014`) partially repairs `ALG-0005` overfitting by detecting common-boundary parallel and optional-singleton-parallel blocks before exact automaton fallback. EXP-0016 shows balanced held-out parallel improves from 0/4 to 4/4 and `optional_singleton_parallel_branch` is fixed, but prefix-biased held-out parallel remains 0/4 and rare reversal noise can be misclassified as a parallel block.
- For `ALG-0014`, exact fallback successes on loops and duplicate labels are memorization, not structural generalization. Future results must distinguish selected grammar from fallback automaton behavior.
- The support-guarded prefix block miner (`ALG-0015`) adds support-skew checks, prefix-bias merge, and same-activity-set dominant-sequence handling. EXP-0017 fixes prefix-biased held-out parallel and rare-reversal noise precision, but the noise fix intentionally replays only the dominant 3/4 traces.
- The grammar-only ablation (`ALG-0016`) shows which `ALG-0015` successes are true grammar behavior: prefix-biased/balanced parallel, rare-reversal dominant sequence, simple parallel, sequence, and optional-singleton parallel. Skip, XOR, loops, duplicate labels, optional chains, and optional concurrency still depend on exact fallback.
- EXP-0018 attributes `ALG-0015`'s feature effects: support-skew alone rejects unsupported noisy parallel grammar but exact fallback still memorizes noise; prefix merge is causal for prefix-biased held-out parallel; dominant-sequence handling is causal for rare-reversal precision. `ALG-0015` is now `deep-testing`, while `ALG-0017`, `ALG-0018`, and `ALG-0019` are ablation controls.
- The current dominant-sequence policy is conservative under incomplete parallel evidence: when only one order is observed, it selects sequence behavior and rejects held-out reversed order.
- EXP-0019 shows prefix merge has an identifiability ambiguity: the same training traces `A B C D E` and `A B D C E` can mean full B/C/D parallel or B followed by C/D parallel. `ALG-0015` chooses full-parallel generalization; `ALG-0020` chooses sequence-prefix precision.
- For noisy reversal evidence, support skew is a major policy knob: `max_parallel_support_skew=2` treats 2:1 reversal evidence as parallel, while skew 1 treats it as noise through dominant-sequence selection. Dominant-ratio 85 is too strict for 3/4 and 5/6 toy noise cases.
- The ambiguity-aware prefix-block PMIR refinement (`ALG-0021`) preserves both common-boundary and prefix-merge interpretations when both are supported and differ. It currently compiles the same selected net as `ALG-0015`, so the annotation is evidence for a future selector or multi-net protocol, not a selected-net quality improvement by itself.
- The ambiguity-set protocol (`ALG-0022`) compiles `ALG-0021` alternatives into separate Petri nets. EXP-0021 shows the set can preserve both full-parallel generalization and sequence-prefix precision, but the event log alone still does not choose between them.
- The cut-limited single-rework loop miner (`ALG-0023`) fixes the repeated-anchor loop pattern when both zero-iteration and one-iteration traces are observed. It uses duplicate visible transitions for the repeated anchor label and generalizes to additional loop iterations, so bounded-count loop behavior remains an explicit risk.
- EXP-0023 confirms `ALG-0023`'s loop assumption: it works with prefix/suffix sequence context, but bounded-at-most-one, multi-body loop choice, nested loop-choice, and one-iteration-only evidence are outside the current scope.
- The cut-limited multi-body rework-loop miner (`ALG-0024`) extends the `ALG-0023` loop line to two or more observed singleton rework bodies around the same anchor. EXP-0024 shows it replays held-out body combinations in multi-body and nested-context loop-choice cases, but it still uses an unbounded-repeat prior.
- Bounded-count loop semantics are not identifiable from zero/one loop evidence alone in the current framework. Treat at-most-one versus unbounded repeat as an explicit prior, selector, or model-set issue rather than an automatic selected-net improvement.
- EXP-0025 shows `ALG-0024` is stable under checked trace-order permutations, but remains singleton-body-only, duplicate-label-conservative, and unbounded-repeat by prior. Length-2 body loops, duplicate body/suffix labels, bounded-count priors, and rare-body noise policy need separate candidates or explicit selectors.
- The cut-limited length-bounded rework-loop miner (`ALG-0025`) extends `ALG-0024` to body alternatives of length at most two. EXP-0026 shows it fixes length-2 and mixed singleton/length-2 body-loop held-out combinations while preserving singleton-body behavior, but bounded-count priors, length greater than two, duplicate-label body contexts, one-iteration-only evidence, and support/noise body selection remain separate questions.
- The loop count policy-set protocol (`ALG-0026`) treats bounded-count loop semantics as a model-set ambiguity: from the same zero/one loop evidence it preserves both `unbounded_repeat` and `at_most_once` Petri-net alternatives. EXP-0027 shows the alternatives bracket repeated-iteration held-out replay versus bounded-count negative rejection, but no final selector exists yet.
- The loop count validation selector (`ALG-0027`) chooses among `ALG-0026` alternatives only when explicit validation positives/negatives uniquely identify one policy. EXP-0028 shows it selects both unbounded-repeat and at-most-once on discriminating validation and returns unresolved, validation-inconsistent, or no-policy-set otherwise. EXP-0029 promotes it to `promising` after frozen train/validation/final-test protocol cases pass and validation replay proxy counts are added; it should not move to `deep-testing` until upstream one-iteration-only, duplicate-label, length >2, and rare-body/noise limits are tested.
- EXP-0030 confirms `ALG-0027`'s current scope boundary: one-iteration-only, duplicate-label, length >2, and overlapping-body-label evidence emit no policy set, while rare-body/noise validity cannot be solved by a loop-count selector. Treat rare loop-body support/noise as a separate candidate line.
- The body-support guard (`ALG-0028`) filters singleton rare loop bodies only under a 3-count and 75-percent dominant-body prior. EXP-0031 shows it improves precision on rare-body-as-noise probes but drops valid rare behavior under the same 3:1 support pattern, so it remains `smoke-tested` and must be described as a precision prior rather than an identifiable discovery claim.
- EXP-0032 confirms `ALG-0028` support thresholds are a policy knob: 2:1 rejects 15/17 rare-noise probes but replays 0/10 valid-rare positives, while 5:1 replays 8/10 valid-rare positives but rejects only 4/17 rare-noise probes. `ALG-0029` adds explicit validation selection for loop-body inclusion and should be treated like `ALG-0027`: useful only under declared validation assumptions, not as training-log-only discovery.
- EXP-0033 confirms `ALG-0029` can pass split validation/final controls for keep-all versus support-guarded body inclusion, including leakage detection. It still cannot resolve mixed rare-body cases where one rare body is valid and another is noise, nor rare bodies with count two under the current support-guard scope.
- `ALG-0030` treats loop-body inclusion and loop-count semantics as separate validation-scoped axes. EXP-0033 shows the four keep/filter x unbounded/at-most-once product quadrants work on controlled cases; EXP-0038 adds shared product totals, but this remains a validation-scoped selector protocol, not a training-log-only discovery claim.
- EXP-0034 broadens `ALG-0030` product evidence to length-2 and mixed-width bodies: all tested width/product quadrants pass, but duplicate-label, overlapping-label, length >2, and one-iteration-only evidence remain upstream blocked cases.
- `ALG-0031` is the rare-count-two support-guard ablation. EXP-0034 shows configured count-two filtering can work under validation, but it still filters rare bodies as a group and cannot handle one valid plus one noisy rare body with the same count.
- `ALG-0032` is the per-body rare inclusion validation selector. EXP-0035 shows bounded per-body keep/drop vectors can repair mixed valid/noisy rare-body cases for count-one and configured count-two policies, while partial validation signal, validation conflicts, weak dominance, and rare-body cap overflow remain explicit non-selection statuses.
- Per-body rare inclusion is not identifiable from the training log alone in the current framework. Treat `ALG-0032` as a validation-scoped selector with a bounded alternative cap; selector outputs carry both naive all-alternative totals and shared totals, but primitive-level shared instrumentation is still missing.
- EXP-0036 broadens `ALG-0032` with split validation/final controls: singleton, length-2, mixed-width, and configured count-two rare-body cases select correctly; final-only signal remains unused; validation/final and train/final leakage are flagged by the harness; duplicate/overlapping label cases remain upstream blocked-scope controls.
- Shared accounting convention for validation-scoped loop selectors: charge one upstream `ALG-0025` discovery once, charge each guarded body alternative by `max(0, alternative_total - base_total)`, charge all compiled count-policy alternatives for `ALG-0030` selector evaluation, and keep validation replay proxy costs per alternative.
- EXP-0037 confirms the `ALG-0032` cap boundary is still a durable operation-cost warning after removing double-counting: allowing three rare bodies creates eight alternatives; the cap-three case falls from 7430 naive operations to 3167 shared operations, still above the deep soft budget for that toy log.
- EXP-0038 integrates the shared-accounting convention into `ALG-0029`, `ALG-0030`, and `ALG-0032` selector outputs while retaining naive totals; the shared-cost report now cross-checks integrated fields rather than serving as the only source of shared totals.
- EXP-0039 cap stress confirms `ALG-0032` exhaustive per-body enumeration exceeds the deep soft budget from two rare bodies onward even with shared accounting; cap-plus-one refusals remain cheap and are part of the limited-operation behavior, not just a defensive error status.
- `ALG-0033` is the direct-signal lower-cost split from `ALG-0032`: validation positives require rare bodies to be kept, validation negatives form rare-body drop clauses, required-keep bodies can unit-propagate singleton drop signals, and at most one guarded net is compiled. EXP-0040 shows this avoids exhaustive `2^R` enumeration and stays under `B_deep` through a five-rare-body direct-signal synthetic family, but non-unit ambiguous negatives remain unresolved by design.
